Decision history for the canvas map

The canvas feature made a small set of choices that keep GeoJSON intake strict, projection work server-side, and map interaction centered on one SVG camera model. This document captures those choices, the alternatives considered, and the reasons each path was accepted or rejected.

GeoJSON boundary Effect Schema validation d3-geo projection SVG viewBox camera Bounds-based focus

The accepted design treats the SVG viewBox as the camera. Every focus, pan, and zoom operation produces a new Bounds object and lets SVG render the correct coordinate rectangle.

Decision timeline

These decisions were clarified while reviewing the implementation and reasoning about how the canvas should parse, validate, project, focus, and move.

Coordinate contract

Accept positions with longitude, latitude, and optional numeric dimensions because GeoJSON positions can carry extra coordinate values. Projection consumes only the first two numbers.

GeoJSON tooling

Keep JSON parsing simple, use Effect Schema for runtime validation, rely on d3-geo for projection, and defer heavier geospatial libraries until the feature needs geospatial analysis.

State focus

Focus a state by deriving a new camera rectangle from the selected state's bounds, not by zooming the default camera in place.

Camera model

Keep the full state map static. Use the SVG view box as the movement model only after a state is selected and the district layer is mounted.

District drag target

Let district-mode drag panning start from the whole SVG surface, not only the selected state path, because an isolated state can leave useful camera space around the shape.

Reset timing

Mount the full state set during reset, reveal the non-selected states with opacity while the camera animates back to the full-map frame, then clear selection after the animation completes.

Accepted decisions

The feature keeps its important choices small. Each one reduces the number of places where coordinate conversion, validation, or camera state can diverge.

Accept GeoJSON dimensions [number, number, ...number[]] matches valid positions.
Validate at boundary Effect Schema checks unknown JSON before projection.
Project on server The browser receives compact SVG paths, bounds, and districts grouped by state.
Use district-mode camera The full map selects states; the selected district map owns zoom and pan.

Decision: allow optional coordinate values

The position type is wider than the projection currently needs because valid GeoJSON positions are not limited to exactly two numbers.

Accepted

Current design Type contract

Model a coordinate as readonly [number, number, ...number[]]. The first two values are longitude and latitude. Extra values remain valid input. 

export type Position = readonly [number, number, ...number[]];

Rejected

Too narrow Exact tuple

A strict [number, number] tuple would reject GeoJSON that includes an altitude or another numeric dimension, even though the feature can safely ignore it. 

// Rejected because it is narrower than GeoJSON positions.
type Position = readonly [number, number];

Runtime behavior

Projection input 2D render

The projection deliberately reads only longitude and latitude. Optional dimensions are accepted at the boundary but do not influence the SVG map. 

const point = projection([position[0] ?? 0, position[1] ?? 0]);

Decision: keep GeoJSON handling narrow

The implementation does not try to become a general geospatial system. It accepts the subset of GeoJSON needed by the map assets and chooses libraries by responsibility.

Library and responsibility choices
Need Chosen path Reason Deferred alternative
Parse GeoJSON Bun.file(...).json() GeoJSON is JSON, so native JSON parsing is enough for local assets. A parser package would add little value here.
Validate runtime shape Effect Schema The project already uses Effect and the feature only accepts a narrow boundary subset. ajv with GeoJSON JSON Schema for broader standards validation.
Type GeoJSON values Local feature types, with @types/geojson available The feature contracts are intentionally shaped around the render model and supported geometry. Import more standard GeoJSON types if the feature grows beyond this subset.
Project coordinates d3-geo It directly supports the Mercator projection and fits the SVG rendering problem. None needed for current projection work.
Geospatial analysis Not included yet The feature currently needs projection, paths, and bounds, not containment, area, or topology operations. @turf/turf or selected Turf modules when those operations become product requirements.
Why not use a full GeoJSON validation library now?

A generic validator can confirm standard GeoJSON shape, but this feature needs a product-specific boundary: feature collections with Polygon and MultiPolygon only, loose properties, and specific name keys used later by the renderer. The local Effect schema expresses that exact boundary and keeps bad or unexpected files from reaching projection code.

Decision: focus states by changing the viewBox

State focus changes the SVG camera rectangle and switches the canvas into district inspection mode. It does not enable zoom around the default full-map center.

Why this is needed

  • District mode needs scale and position. A smaller view box isolates the selected state. A different minX and minY moves the camera to that state.
  • Default-center zoom is wrong for edge states. Keeping the full-map center would zoom into the middle of the map, not necessarily the clicked state.
  • Bounds give the target rectangle. viewBoxForBounds uses selected state bounds, aspect ratio, and coverage to build the focused camera.

Code path

const targetViewBox = viewBoxForBounds(
  state.bounds,
  aspectRatio,
  window.matchMedia("(max-width: 640px)").matches ? 0.76 : 0.84,
);

zoomBaseViewBoxRef.current = targetViewBox;
animateViewBox(targetViewBox);

The selected-state camera becomes the district zoom baseline. Full-map pan and zoom remain disabled until a state is selected.

Decision: reject CSS scale and scroll as the camera

Scaling and scrolling could produce a visible zoom effect, but it would move the DOM viewport instead of the SVG map camera.

Accepted camera model

One source of truth SVG coordinates

Store one Bounds value as viewBox. State mode uses it to frame India; district mode updates it for selected-state pan and zoom. 

viewBox: Bounds

viewBox -> "minX minY width height"

Rejected movement model

Extra coordinate layer Pixel scroll

CSS scale plus scroll would require converting projected SVG coordinates into layout pixels, then keeping scroll offsets, transform origins, hit testing, wheel behavior, and keyboard movement aligned. 

// Rejected as primary model:
content.style.transform = "scale(...)";
scrollContainer.scrollTo(pixelX, pixelY);

Still needs bounds

No real shortcut State center

Centering a selected state still requires knowing the state bounds. Once those bounds exist, using them directly as an SVG camera target is simpler than routing district inspection through CSS pixels and scroll containers. 

state.bounds -> viewBoxForBounds(...)
state.bounds -> pixel scroll math // avoid

Consequences

The accepted path makes some behavior easier and leaves a few deliberate constraints in place.

Tradeoffs from the current decisions
Area Benefit Constraint
Validation Unknown file data is checked before path generation. The accepted GeoJSON subset must be updated if new geometry types are needed.
Projection States and districts share one projection, so layers align. The server prepares paths before hydration, so file changes require a new server load.
Interaction Selection is static in full-map mode; pointer, wheel, and keyboard movement are scoped to district mode. District camera math must stay in SVG coordinate space, background drags must not reset the map after panning, and reset should fade the full state set in before clearing selection.
Accessibility Status text and state labels reflect the same selected camera state. Map-only behavior still needs structured alternatives as the product grows.
Core rule that emerged

Keep data validation at the boundary, keep projection output serializable, and keep district camera movement in SVG coordinates. That rule prevents parsing, rendering, and interaction code from each carrying its own version of map state.

Code anchors

These files hold the implementation points behind the decisions recorded here.

types.ts

Contracts

Defines Position, Bounds, CanvasMapFeature, and CanvasMap. This is where optional coordinate dimensions and the serializable render model are expressed.

schemas.ts

Runtime validation

Uses Effect Schema to validate the supported GeoJSON subset. The position schema accepts longitude, latitude, and optional numeric dimensions.

utils.ts

Projection Camera math

Converts validated boundaries into SVG paths and bounds, then exposes helpers such as viewBoxForBounds, constrainViewBox, and translateBounds.

components/canvas.tsx

Presentation Interaction

Stores the active view box, switches between state and district modes, handles district-only camera input from the whole selected-state surface, lazily renders district layers, and announces user-visible status.
