How the canvas feature becomes an interactive India map

The canvas feature starts with Survey of India GeoJSON files on disk. It validates them on the server, projects boundaries into SVG coordinates, serializes a compact render model through the home route loader, then lets the client component switch between a static state-selection map and a movable selected-state district map.

End-to-end flow

The feature has five major handoff points. Each point reduces uncertainty before the next layer uses the output.

1 Local map files

Bun.file(...).json() reads state and district GeoJSON from public/maps/survey-of-india.

2 Effect schema boundary

Schema.decodeUnknownSync proves the JSON is a feature collection with supported geometry.

3 Projection utilities

prepareMap creates a Mercator projection, path strings, feature bounds, and the padded base view box.

4 Route loader

loadCanvasMap returns { renderMap }. The home route passes it directly into <Canvas map={...} />.

5 Interactive SVG

The client first renders only states. After selection it isolates one state, mounts that state's district layer, enables district pan/zoom, and announces status.

Data sources

The starting point is not a network call or database query. It is two local GeoJSON files loaded by a server-only service.

Boundary input contract

  • BoundaryFeatureCollection A GeoJSON feature collection with a readonly feature list.
  • BoundaryFeature A feature with a loose property bag and supported geometry.
  • BoundaryGeometry Only Polygon and MultiPolygon are allowed.
  • Position At least longitude and latitude, with optional extra numeric values.

Validation behavior

BoundaryFeatureCollectionSchema mirrors this narrow GeoJSON subset. The decoder is synchronous because the service has already awaited file reads and fails fast before projection or path generation can run on malformed data.

Server path

Server code does the expensive and risky work: file IO, runtime validation, projection setup, and SVG path generation. This keeps the browser payload compact and avoids client-side GeoJSON parsing.

loadCanvasMapService

src/features/canvas/server/canvas.service.ts
  • Starts both file reads with Promise.all.
  • Loads state and district JSON through Bun file APIs.
  • Decodes both unknown JSON values with the Effect schema.
  • Calls prepareMap(states, districts).
  • Returns { renderMap } for the route loader.

prepareMap

src/features/canvas/utils.ts
  • Builds one Mercator projection from state boundaries.
  • Renders each state with the STATE property key.
  • Renders each district with the DISTRICT property key.
  • Groups districts by the matched state name, including the Andaman source alias.
  • Merges rendered state bounds into a full map extent.
  • Adds 18 SVG units of padding to produce baseViewBox.

createMapProjection

src/features/canvas/utils.ts
  • Uses geoMercator().scale(1).translate([0, 0]) as a measuring pass.
  • Projects every state coordinate into temporary bounds.
  • Fits the projected bounds inside a 1000 unit SVG coordinate space.
  • Applies 24 units of internal padding.
  • Returns a final projection reused by both state and district layers.

renderFeature

src/features/canvas/utils.ts
  • Delegates geometry conversion to renderGeometry.
  • Extracts the display name with propertyString.
  • Falls back to Boundary {index + 1} when the name is missing.
  • Builds a stable serializable feature object.

renderGeometry

src/features/canvas/utils.ts
  • Normalizes Polygon into a one-item polygon list.
  • Iterates each ring and skips empty rings.
  • Projects the first point into an Mx,y command.
  • Projects remaining points into Lx,y commands.
  • Closes each ring with Z and returns bounds plus path.

loadCanvasMap

src/features/canvas/server/canvas.queries.ts
  • Defines a TanStack Start server function with method GET.
  • Acts as a thin route-facing query wrapper.
  • Keeps service details out of route files.

Render model

The output of server preparation is a serializable model tailored to SVG rendering and camera math.

CanvasMapFeature

type CanvasMapFeature = {
  id: string;
  name: string;
  path: string;
  bounds: Bounds;
};

The path is ready to assign to an SVG <path d="...">. The bounds let the client focus a state without retaining original GeoJSON coordinates.

CanvasMap

type CanvasMap = {
  states: CanvasMapFeature[];
  districtsByState: Record<string, CanvasMapFeature[]>;
  baseViewBox: Bounds;
};

This object is passed through TanStack Start serialization. Districts remain render-ready, but are grouped so the client mounts only the selected state's layer.

Route handoff

The route files remain entry points. They load data, compose the page, and avoid owning map transformation logic.

src/routes/__root.tsx

Adds the root document and app scripts only. Browser zoom shortcuts are not intercepted before hydration; hydrated district mode owns its own canvas zoom shortcuts.

src/routes/index.tsx

Defines the home route. Its loader calls loadCanvasMap(). The component reads renderMap from Route.useLoaderData() and renders <Canvas map={renderMap} />.

Canvas component boundary

Receives a fully prepared CanvasMap. From this point forward, the code is about interaction state and presentation, not data loading.

Client runtime

The component is marked "use client". It renders the map and manages a small runtime state machine around camera, interaction, and announcements.

React state

  • viewBox The visible SVG camera. Converted to the SVG viewBox string.
  • selectedState State name that switches the canvas from full state selection into district inspection mode.
  • focusedState State name used to visually mirror keyboard focus.
  • isDragging Switches the map cursor between grab and grabbing.
  • status Accessible status text used by the SVG label and screen-reader live region.

Refs for fast interaction

  • viewBoxRef Latest camera value for selected-state wheel, key, drag, and animation calculations.
  • zoomBaseViewBoxRef Active camera baseline: full map in state mode, selected-state fit in district mode.
  • dragStateRef Pointer id, movement history, and drag threshold for district-map panning.
  • cameraAnimationRef Current animation frame id so direct actions can cancel in-flight tweens.
  • suppressNextClickRef Prevents a drag-generated click from resetting or selecting unexpectedly.
How the component presents the render model

The SVG root receives the computed viewBox string. A background rectangle fills the active camera area. State features render first as interactive paths with role="button", labels, keyboard handlers, focus handlers, fill colors, and selection strokes. District features are not mounted in the initial state map. After a state is selected, only that state remains visible and its district layer renders in a pointerEvents="none" group.

Zoom controls sit in a fixed bottom-right control group only in district mode. The percentage label is a live status. A separate screen-reader-only paragraph announces state selection, district panning, reset, and district zoom changes.

Interaction processors

All user interactions end by updating the same camera model: a constrained bounds object rendered through the SVG viewBox.

Outputs to processors and users

The feature emits outputs at three levels: data objects for downstream code, DOM attributes for SVG rendering, and interface feedback for users.

Server output

{ renderMap: CanvasMap }
  • Consumed by the TanStack Start loader result.
  • Serializable into the SSR payload.
  • Contains no GeoJSON coordinates after projection.

Render output

SVG nodes and attributes
  • viewBox changes create district-mode pan and zoom.
  • State d attributes draw boundaries.
  • Grouped district d attributes draw non-interactive overlays only after selection.
  • Stroke width and color represent focus or selection.

Accessible output

Labels and live regions
  • The canvas section has role="application" and keyboard shortcuts.
  • The SVG role is img, labelled by current status.
  • Each state path has a button role and state name.
  • Status updates are mirrored in an aria-live="polite" region.
Important implementation choice

baseViewBox still frames the full state-selection map. Once a state is selected, the selected state's fitted view box becomes the district zoom baseline and pan extent, so users cannot drag the district map into the rest of India.

Responsibility boundaries

The feature follows the project architecture by keeping route files thin, map preparation in server-side modules, reusable math in pure utilities, and interaction state inside the client component.

Server owns

  • Local file IO through Bun.
  • Runtime schema validation with Effect.
  • Projection and SVG path generation.
  • Construction of the serializable grouped render model.

Client owns

  • SVG presentation of prepared paths.
  • State-selection mode and selected-state district mode.
  • District-only camera state, zoom percentage, panning, and lazy district rendering.
  • Visible selection, Escape reset, and accessible status updates.
